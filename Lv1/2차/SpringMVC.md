# Spring MVC
- Spring Framework에서 웹 앱을 위해 제공되는 서블릿 API기만 프레임워크
#### Servlet(서블릿)
- 클라이언트의 요청을 특정 규약(약속)에 맞춰서 Java코드로 작성하는 클래스 파일
- 아파치 톰캣: 서블릿을 웹앱으로 실행할 수 있도록 해주는 서블릿 컨테이너이다

### MVC
- 앱을 개발할 때 사용하는 디자인 패턴
- Model, View, Controller로 구분해서 각 역할에 맞게 코드를 작성하는 개발 방식
#### Model:
클라이언트의 요청을 처리해서 응답을 돌려줄때 돌려주는 <b>작업의 처리 결과와 데이터를 Model</b>이라고 한다

#### View:
Model을 이용해서 웹 브라우저와 같은 <b>앱의 화면에 보이는 리소스(자원)를 제공하는 역할</b>을 한다
- HTML 페이지 출력
- PDF, Excel 등의 문서 형태로 출력
- XML, JSON 등 특정 형식의 포맷으로 변환

#### Controller:
클라이언트의 요청을 직접적으로 받는 엔드포인트(끝부분)로써 <b>Model과 View 중간에서 상호작용</b>을 한다

### Spring MVC의 동작 순서

- 요청 발생
- DispatcherServlet(요청을 어디로 보내고, 어떻게 처리할지 조정하는 중앙 컨트롤러)에서 모든 요청을 가로챔 
- 핸들러(요청을 실제로 처리할 메서드/함수) 조회 : 핸들러 매핑(요청 URL을 어떤 핸들러가 처리할지 찾기)된 핸들러를 조회한다
- 핸들러 어댑터(핸들러 실행기) 조회 : 핸들러 어댑터 찾기
- 핸들러 어댑터 실행 : 핸들러 어댑터 실행
- 핸들러 실행 : 핸들러 어댑터로 핸들러 실행
- ModelAndView(Model과 View을 함께 담아서 반환하는 객체) 반환 : 핸들러가 반환하는 정보를 모델과 뷰로 변환해서 반환
- viewResolver(뷰 이름을 실제 화면 파일로 바꿔주는 역할) 호출 : 뷰 리졸버를 실행한다
- View 반환 : 뷰 리졸버는 뷰의 논리적이름을 물리적 이름으로 바꾸고 랜더링 역할을 하는 뷰 객체로 반환
- 뷰 렌더링 : 뷰를 통해 뷰 파일을 실행해서 최종 HTML로 만든다
